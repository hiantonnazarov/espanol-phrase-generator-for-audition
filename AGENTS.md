# Espanol audio

Репозиторий хранит:
- `1_vocabulary_raw.md` — исходный «грязный» конспект.
- `2_vocabulary.md` — производный очищенный словарь и грамматическую шпаргалку.
- `3_topics.md` — порядок учебных тем и прогрессию сложности.
- `5_phrases.md` — основной редактируемый файл с учебными фразами в читаемом Markdown-формате.
- `5_phrases_audio.md` — производный очищенный файл тех же фраз для аудио/плоского текста; вручную не редактируется.
- `4_words_usage.md` — производный отчет покрытия словаря фразами.

## Scripts

- `scripts/build_vocabulary.ps1` — основной вход для пересборки `2_vocabulary.md`.
- `scripts/build_vocabulary_sectioned.ps1` — фактический генератор `2_vocabulary.md` из `1_vocabulary_raw.md`.
- `scripts/build_phrases_audio.ps1` — генератор `5_phrases_audio.md` из `5_phrases.md`.
- `scripts/build_words_usage.ps1` — генератор `4_words_usage.md` по `2_vocabulary.md` и `5_phrases.md`.
- `scripts/build_incremental_update_report.ps1` — генератор инкрементального отчета по новым элементам словаря и их покрытию.
- `scripts/test_build_phrases_audio.ps1` — тест генератора `5_phrases_audio.md`.
- `scripts/test_build_vocabulary.ps1` — smoke-тест пересборки словаря.
- `scripts/test_build_words_usage.ps1` — тест генератора отчета по фразам.
- `scripts/test_build_incremental_update_report.ps1` — тест генератора инкрементального отчета.

## Commands

```powershell

.\scripts\build_words_usage.ps1

.\scripts\test_build_words_usage.ps1
.\scripts\test_build_incremental_update_report.ps1
```

## Structure Rules

- Все рабочие и тестовые скрипты держать только в папке `scripts`.
- Придерживаться принципа `1 шаг = 1 файл`: один основной build-скрипт на одно действие и один отдельный test-скрипт на его проверку.
- Не поддерживать параллельно дублирующие реализации в `Python`, `legacy` или `fast`, если это не единственный рабочий путь.
- Если по ходу работы обнаруживается повторяющаяся проблема или полезное правило, его нужно закреплять в `AGENTS.md`; если правило можно проверять или применять автоматически, его нужно отражать и в соответствующих скриптах из `scripts`.

## Build Rules For `2_vocabulary.md`

- Брать данные из `1_vocabulary_raw.md` и собирать новый чистый `2_vocabulary.md`.
- Убирать дубли, ссылки, картинки, фонетический мусор, случайные обрывки таблиц и прочие технические артефакты.
- Сохранять только содержательные словарные и грамматические данные.
- Группировать лексику по смыслу: глаголы, местоимения, состояния `estar/tener`, предлоги, вопросительные слова, наречия, числительные, время, погода, дом, природа, фразы.
- Для словарных разделов использовать Markdown-таблицы ровно из двух колонок: `Испанский | Перевод на русский`.
- Для грамматических разделов разрешены отдельные специализированные таблицы по временам и правилам.

## Build Rules For `5_phrases.md`

- Строить `5_phrases.md` по темам строго в порядке из `3_topics.md`.
- Для каждой темы использовать отдельный раздел Markdown и таблицу ровно из двух колонок: `Испанский | Перевод`.
- Внутри темы ДО таблицы фраз разрешён блок «общая инфа по теме» (правила, списки, формы, заметки). Этот блок — часть учебного материала: его нельзя удалять/переписывать скриптами, и его не нужно учитывать в `4_words_usage.md` (в учёт идут только фразы в конце темы).
- Таблица фраз может быть как с заголовком/разделителем, так и «без шапки» (просто строки `| Испанский | Перевод |`); парсеры должны считать фразами только финальный блок фраз темы, а не таблицы из общей инфы.
- Фразы должны быть живыми и учебными, а не набором отдельных слов.
- Раздел `12. Числительные` не обязателен: количественные и порядковые числительные не нужны в `5_phrases.md` и могут отсутствовать.
- Средняя длина фраз должна быть примерно 4-7 слов, но допустимы небольшие отклонения ради естественности.
- В ранних темах держать фразы проще и не забегать далеко вперед по грамматике; по мере продвижения постепенно усложнять материал.
- По возможности использовать лексику и формы из `2_vocabulary.md`; новые слова добавлять умеренно и только когда без них фраза звучит неестественно.
- Избегать сильных повторов внутри одной темы, если можно взять соседние слова или формы из словаря.
- Фразы должны покрывать ключевые слова и формы темы, кроме очевидно массовых наборов вроде всех числительных.
- В узких тематических разделах каждая фраза должна явно содержать материал самой темы, а не только общий фон. Примеры: в темах по временам каждая фраза должна содержать форму соответствующего времени, в теме по календарю — месяц или день недели, в теме по времени и датам — формулу времени или даты, в погодных темах — погодную лексику.
- Если `12. Числительные` всё же оставлен, держать компактный список в формате `слово | перевод` (без `=` и слешей); полноценные предложения не требуются.
- В темах по временам каждая фраза должна содержать не только целевую форму, но и хотя бы одно естественное указание времени: `ahora`, `hoy`, `mañána`, `ayér`, часть суток, день, месяц, сезон или другой уместный временной маркер.
- Внутри одного временного раздела не залипать на одном и том же маркере вроде `hoy`, `ahora`, `mañána`, `ayér`, `ántes`, если словарь уже даёт естественные варианты (`esta mañána`, `esta tárde`, `esta nóche`, `después`, `luego`, `hásta hoy`, `anteayér`, `de repénte`, сезоны, дни недели и т.п.).
- Русские переводы в `5_phrases.md` должны быть естественными для живого русского, а не дословными кальками с испанского. Если буквальный перевод по-русски звучит неестественно, его нужно перестраивать в нормальную русскую формулировку.
- Не оформлять строки в `5_phrases.md` как словарные пояснения вида `X = Y`; вместо этого писать полноценные учебные предложения.
- Не держать варианты через слеши или обратные слеши внутри одной фразы, если их можно развернуть в отдельные строки.
- `5_phrases.md` является каноническим редактируемым файлом; любые содержательные правки вносятся только сюда.
- После правок `5_phrases.md` нужно пересобирать `5_phrases_audio.md`, а не редактировать его вручную.
- При следующем ребалансировании в приоритете должны быть элементы словаря, которые ещё не использованы.
- При ребалансировании однокоренные глаголы по возможности использовать только по одному разу на корень; несколько форм одного и того же корня добавлять только когда это нужно для покрытия грамматической темы.
  - Для глаголов не дублировать один и тот же смысловой пример только ради перебора лиц. Если глагол уже покрыт в нужном времени, дополнительные лица добавлять только когда они реально нужны для суммарного покрытия лиц по разделу или для новой лексики.
  - В темах по временам достаточно, чтобы каждый глагол встретился минимум один раз в нужном времени, а лица были покрыты суммарно по разделу; не нужно держать один и тот же глагол во всех лицах.
- То же правило распространяется и на `1. Presente`: для каждого глагола в разделе достаточно одной формы, а лица должны быть покрыты суммарно по разделу, а не по каждой лемме отдельно.
- При ручном ребалансировании `1. Presente` не держать стартовый блок фраз почти целиком на `yo`, если те же леммы можно естественно распределить по `tú`, `él/ella`, `nosotros`, `vosotros`, `ellos` без потери покрытия.
- То же правило распространяется и на остальные временные разделы (`Pretérito perfecto`, `Imperfecto`, `Pretérito perfecto simple`, `Futuro simple`): не держать раздел или его стартовый блок с явной концентрацией на `yo`, если те же леммы и маркеры времени можно естественно распределить по другим лицам.
- При ребалансировании наречий места не использовать `aquí` как дефолтную заглушку, если по смыслу так же естественно подходят `ahí`, `allí`, `allá` или `acá`.
  - Раздел `25. Фразы: разговорное/кафе/быт` считать резервным хвостом: если фраза естественно относится к более ранней теме, её нужно переносить туда, а не оставлять в разделе `25`.
  - При удалении дублей из раздела `25` приоритет всегда у более ранней тематической темы, если состояние, лексема или формула уже покрыты выше.

## Build Rules For `5_phrases_audio.md`

- `5_phrases_audio.md` строится автоматически из `5_phrases.md` скриптом `scripts/build_phrases_audio.ps1`.
- Сохранять порядок разделов и пустых строк из `5_phrases.md`.
- Убирать Markdown-оформление, табличные разделители, обратные слеши, служебные маркеры форматирования и прочий шум, мешающий плоскому тексту / аудио.
- Не менять смысл, порядок фраз, переводы и учебный материал; допустима только техническая очистка представления.
- Не использовать `5_phrases_audio.md` как источник для `4_words_usage.md` или инкрементального отчета.

## Build Rules For `4_words_usage.md`

- Пересчитывать `4_words_usage.md` автоматически из `2_vocabulary.md` и `5_phrases.md`.
- Единица учета — каноническая словарная единица: одиночное слово или многословное выражение из словарных разделов после отбрасывания legacy-агрегатов и справочных таблиц.
- Для одиночных слов считать целые совпадения слова, чтобы не ловить случайные подстроки.
- Для многословных записей считать вхождение нормализованной фразы целиком (без учета регистра и учебных ударений на гласных).
- Если многословная запись после удаления служебных токенов (артикли/простые предлоги) сводится к одному смысловому слову длиной 4+, учитывать использование по этому слову (чтобы вариации артиклей/предлогов не шумели в отчете).
- В отчете обязательно держать три блока: покрытые элементы с числом использований, неиспользованные элементы, потенциально перегруженные элементы.
- В отчете отдельно показывать канонически неучитываемые legacy-элементы.
- В отчете отдельно показывать покрытие глаголов по временам и покрытие лиц по временным разделам.
- В отчете отдельно показывать кандидатов на сокращение фраз, если удаление строки не ломает каноническое покрытие, покрытие глагола по времени, покрытие лиц по временному разделу и тематическую привязку.
- В отчете дополнительно держать машинную проверку соответствия фраз своей теме для узких разделов: времена, календарь, время и даты, погода, природа.
- В отчете дополнительно держать машинную проверку типичных дословных русских переводов, чтобы ловить явные кальки и неестественный порядок слов.
- В отчете дополнительно держать машинную проверку временного контекста для тем по временам.
- В отчете дополнительно держать машинную проверку перекоса лиц в временных разделах, чтобы ловить явную концентрацию фраз на `yo`.
- В отчете дополнительно держать машинную проверку плохого оформления фраз: строки со слешами и псевдо-словарные записи с `=`.
- Не пытаться искусственно балансировать служебные слова вроде артиклей и базовых предлогов, если они шумят в отчете.
- Не считать обязательными единицами покрытия строки из грамматических таблиц с личными формами, служебные примеры из phrasebook-блока и legacy-агрегаты вида `X = Y` или пачки вариантов через слэш.
- Критерий завершения работы над покрытием: все нечислительные канонические элементы словарных разделов из `2_vocabulary.md` должны встретиться в `5_phrases.md` минимум один раз.
- Количественные и порядковые числительные не являются целями покрытия и не должны учитываться в `4_words_usage.md` (отчёт игнорирует группу `Числительные`).

## Spanish Formatting Rules

- В испанских словах использовать нативные буквы `áéíóú`, а не искусственные маркеры ударения.
- Учебные ударения сохранять во всех многосложных испанских словах, даже если в обычной орфографии такое ударение обычно не пишется.
- Учебные ударения должны ставиться на реально ударный слог; нельзя искажать форму слова ударением на неправильной гласной (`trabajó` вместо `trabájo`, `respondó` вместо `respóndo`, `entrá` вместо `éntra` и т.п.).
- Короткие служебные односложные формы не переакцентовывать искусственно, кроме случаев, где ударение уже есть в нормальной записи (`mí`, `tú`, `él`).
- Исправлять очевидные опечатки и грубые ошибки в испанском.
- При ручной проверке текста отдельно смотреть не только на орфографию, но и на грамматическую корректность и естественность испанского: управление глаголов, согласование, выбор `ser/estar/tener`, уместность времени и типичные ложные друзья.

## Content Normalization Rules

- Для существительных по возможности сохранять или нормализовать артикли.
- Для блоков состояний и ощущений разделять конструкции по глаголу: `Estar + ...` и `Tener + ...`.
- Для указательных местоимений и наречий места хранить формы по одной на строку, а не пачками через слэш.
- Для фраз не использовать мусорные подзаголовки вроде `Примеры слов` внутри итогового словаря.
- Не создавать раздел `Прочее`, если в нем остается только мусор.

## Quality Bar

- Русские переводы и формулировки должны быть стилистически выровнены по всему файлу.
- После пересборки нужно проверять, что из `1_vocabulary_raw.md` не пропали содержательные словарные и грамматические блоки.
- Допустимо сознательно не переносить только фонетику, изображения, тренировочные упражнения и дублирующие технические заметки.
- После изменений в `5_phrases.md` нужно сначала пересобирать `5_phrases_audio.md`, потом `4_words_usage.md` и смотреть, не появилось ли лишних перекосов по содержательной лексике.
- Если отчет показывает перегрузку артиклей, предлогов и других служебных форм, это не считается поводом переписывать хорошие учебные фразы.
- Если в процессе работы вручную исправляются совсем плохие переводы, явные кальки, неестественные формулировки, неудачные паттерны покрытия или перекосы в распределении слов, это нужно считать не разовой правкой, а кандидатом в постоянное правило для `AGENTS.md`.
- Если такая правка носит системный характер, нужно обновлять не только текстовые правила, но и генераторы или тесты, чтобы это требование сохранялось в будущих пересборках и ребалансировках.
- При ребалансировании фраз стремиться к более равномерному распределению содержательной лексики по темам и по всему файлу, не допуская ненужной концентрации одних и тех же слов, если словарь позволяет естественные замены.

## Verification Rules

- После изменения логики генерации словаря запускать `scripts/test_build_vocabulary.ps1`.
- После изменения логики генерации `5_phrases_audio.md` запускать `scripts/test_build_phrases_audio.ps1`.
- После изменения логики отчета по фразам запускать `scripts/test_build_words_usage.ps1`.
- После изменения логики инкрементального отчета запускать `scripts/test_build_incremental_update_report.ps1`.
- Перед финальным завершением прохода проверять `4_words_usage.md`: в блоке `Неиспользованные элементы` не должно оставаться нечислительных канонических элементов.
